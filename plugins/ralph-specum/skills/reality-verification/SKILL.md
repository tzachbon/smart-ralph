# Reality Verification

For fix goals: reproduce the failure BEFORE work, verify resolution AFTER.

## Goal Detection

Classify user goals to determine if diagnosis is needed. See `references/goal-detection-patterns.md` for detailed patterns.

**Quick reference:**
- Fix indicators: fix, repair, resolve, debug, patch, broken, failing, error, bug
- Add indicators: add, create, build, implement, new
- Conflict resolution: If both present, treat as Fix

## Project Type Detection (MANDATORY — before Command Mapping)

Before deciding which verification tools to use, determine the project type.
Read from `requirements.md → Verification Contract → Project type` if present.
If not documented, infer from the codebase:

```
Has a UI / browser entry point?
  ├── YES → project type: fullstack (or frontend)
  │         → UI entry points are valid → Playwright MCP eligible
  └── NO  → check further:
            Has HTTP API endpoints (REST/GraphQL)?
              ├── YES → project type: api-only
              │         → NO Playwright → use WebFetch / curl / test commands
              └── NO  → project type: cli or library
                        → NO Playwright → use test commands / build checks only
```

**When project type is `api-only`, `cli`, or `library`:**
- Do NOT load `mcp-playwright.skill.md`, `playwright-env.skill.md`, or `playwright-session.skill.md`
- Do NOT create VE tasks with UI entry points
- Do NOT run `ui-map-init`
- Verify using: test commands, build checks, `curl`/WebFetch for API endpoints

**When project type is `fullstack` or `frontend`:**
- UI entry points are valid → follow the full e2e skill chain
- Load `playwright-env` → `playwright-session` / `mcp-playwright` → `ui-map-init`

**If project type cannot be determined:** emit ESCALATE:
```
ESCALATE
  reason: project-type-unknown
  resolution: add "Project type: api-only | fullstack | cli | library" to
              requirements.md → Verification Contract, then re-run
```

## Command Mapping

Apply ONLY after project type is confirmed.

| Goal Keywords | Project Type | Verification Command |
|---|---|---|
| CI, pipeline | any | `gh run view --log-failed` |
| test, tests | any | project test command |
| type, typescript | any | `pnpm check-types` or `tsc --noEmit` |
| lint | any | `pnpm lint` |
| build | any | `pnpm build` |
| E2E, UI | **fullstack / frontend only** | Playwright MCP browser tools (skip if api-only/cli/library) |
| API, endpoint | any | WebFetch tool / `curl` |

## VE Task Numbering Contract

VE tasks follow a fixed numbering scheme across all specs. This must be consistent
between `reality-verification`, `ui-map-init`, and the tasks generated by `task-planner`:

```
VE0    — ui-map-init (selector map generation)
         Always the first VE task, runs BEFORE any VE1+ tasks
         Only created for fullstack / frontend projects
         Defined in: ui-map-init.skill.md

VE1..N — individual acceptance-criteria verification tasks
         One per user story or acceptance criterion with a UI/API entry point
         Load playwright-session + mcp-playwright (UI) or WebFetch (API)

4.3    — VF: Verify original issue resolved (fix-type specs only)
         Runs AFTER implementation and PR creation (task 4.2)
         References BEFORE state from .progress.md
         Defined in: reality-verification (this skill)
```

**Rule for `task-planner`**: when generating tasks.md for a fullstack spec with VE tasks,
always include `VE0` (ui-map-init) as the first VE task. Never generate VE1+ without VE0.
For api-only/cli/library specs, skip VE0 entirely.

## BEFORE/AFTER Documentation

### BEFORE State (Diagnosis)

Document in `.progress.md` under `## Reality Check (BEFORE)`:

```markdown
## Reality Check (BEFORE)

**Goal type**: Fix
**Reproduction command**: `pnpm test`
**Failure observed**: Yes
**Output**:
```
FAIL src/auth.test.ts
  Expected: 200
  Received: 401
```
**Timestamp**: 2026-01-16T10:30:00Z
```

### AFTER State (Verification)

Document in `.progress.md` under `## Reality Check (AFTER)`:

```markdown
## Reality Check (AFTER)

**Command**: `pnpm test`
**Result**: PASS
**Output**:
```
PASS src/auth.test.ts
All tests passed
```
**Comparison**: BEFORE failed with 401, AFTER passes
**Verified**: Issue resolved
```

## VF Task Format

Add as task 4.3 (after PR creation) for fix-type specs:

```markdown
- [ ] 4.3 VF: Verify original issue resolved
  - **Do**:
    1. Read BEFORE state from .progress.md
    2. Re-run reproduction command: `<command>`
    3. Compare output with BEFORE state
    4. Document AFTER state in .progress.md
  - **Verify**: `grep -q "Verified: Issue resolved" ./specs/<name>/.progress.md`
  - **Done when**: AFTER shows issue resolved, documented in .progress.md
  - **Commit**: `chore(<name>): verify fix resolves original issue`
```

## Test Quality Checks

When verifying test-related fixes, check for mock-only test anti-patterns. See `references/mock-quality-checks.md` for detailed patterns.

**Quick reference red flags:**
- Mock declarations > 3x real assertions
- Missing import of actual module under test
- All assertions are mock interaction checks (toHaveBeenCalled)
- No integration tests
- Missing mock cleanup (afterEach)

## Why This Matters

| Without | With |
|---------|------|
| "Fix CI" spec completes but CI still red | CI verified green before merge |
| Tests "fixed" but original failure unknown | Before/after comparison proves fix |
| Silent regressions | Explicit failure reproduction |
| Manual verification required | Automated verification in workflow |
| Tests pass but only test mocks | Tests verify real behavior, not mock behavior |
| False sense of security from green tests | Confidence that tests catch real bugs |
| Playwright used on API-only repos | Correct tool chosen for project type |
