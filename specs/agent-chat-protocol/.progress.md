# agent-chat-protocol

## Goal

Crear un canal de **chat bidireccional en tiempo real** entre el executor (agente implementador) y el reviewer, basado en filesystem. El objetivo es resolver los 5 gaps de comunicación que nadie documenta en protocolos de agentes: turn-taking, acknowledgment, silencio intencional vs problema, urgencia, y cierre de debate.

Inspirado en el marco **FLOC** (Floor Control for Agent Collaboration), con señales explícitas (`→ OVER`, `→ ACK`, `→ CONTINUE`, `→ HOLD`, `→ STILL`, `→ ALIVE`, `→ CLOSE`, `→ URGENT`) que resuelven cada gap conversacional.

## Context

El sistema ralph-specum ya tiene:
- `spec-executor.md` — agente que implementa tareas secuencialmente
- `external-reviewer.md` — agente revisor paralelo
- `task_review.md` — canal unidireccional formal (FAIL/PASS/WARNING)
- `.ralph-state.json` — estado compartido entre sesiones

El problema: la comunicación es un solo sentido — el revisor escribe FAIL con fix_hint, el executor lo ejecuta ciegamente. No hay canal para debate, explicación de decisiones, o chat proactivo.

## Interview Responses

### Goal Interview
- **Legibilidad humana**: El chat debe ser legible para humanos desde el principio. No raw — todos los agentes y humanos usan formato legible. El humano puede intervenir y su voz es inapelable.
- **Cierre de debates**: No hay cierre de debate permanente. Los roles son de dominio (quien sabe más del tema tiene más peso), no jerárquicos. El revisor tiene mejor rol de dominio para revisar, pero debe escuchar. Si hay posiciones irreconciliables, la verdad empírica es el árbitro — si ambos investigan y ven la evidencia, llegan a acuerdo. No habrá posiciones irreconciliables si hay información clara.
- **Persistencia del chat**: El chat acumula toda la vida del spec. Más adelante puede ser un problema (como logs con rotación). No preocuparnos ahora, pero el diseño debe permitir rotación dinámica cuando el tamaño sea inmanejable.
- **Human intervention**: El humano puede leer e intervenir en cualquier momento. Su voz es siempre inapelable.

## Learnings

- FLOC as a named protocol does not exist in published literature — web search returned no results. The framework was synthesized internally from walkie-talkie protocols, FIPA ACL, and team chat patterns.
- All three major agent protocols (A2A, ACP, MCP) address message routing, NOT conversational floor management — this is a genuine research gap.
- qa-engineer is NOT a chat participant — it runs as sub-agent via Task tool with synchronous VERIFICATION_PASS/FAIL/DEGRADED return values, no independent session.
- The 8 FLOC signals must be synchronized with the existing PENDING mechanism in task_review.md to avoid dual blocking.
- OVER timeout should be measured in tasks (N=1), not polling cycles, since executor doesn't poll.
- Pre-first-message silence = UNKNOWN, not STILL — silence before reviewer's first ALIVE cannot be assumed healthy.
- URGENT cannot interrupt during active qa-engineer delegation — boundary is after Task tool returns.
- Requirement insight: chat.md activation threshold is existence + 1 message, not just existence — preserves backward compat for specs without reviewer
- Atomic write decision: temp file + rename (Option B) chosen over O_APPEND for explicit cross-filesystem robustness
- HOLD is pre-task gate: executor reads chat only at task START, not mid-task — key semantic that prevents executor blocking on HOLD during execution

## Learnings
- Architecture: Chat is shared file + per-agent state files. No central state coordinator.
- FLOC state machine has 4 states: UNKNOWN, ACTIVE, BLOCKED, HELD, ESCALATED
- OVER timeout is 1 task cycle, not polling cycles — executor does not poll chat
- STILL TTL = 3 tasks, resets on ANY signal, triggers ALIVE when would expire
- INTENT-FAIL gives executor 1 task window to correct before FAIL written to task_review.md
- URGENT boundary is after qa-engineer delegation completes (not during Task tool call)
- qa-engineer excluded from chat — has no independent session, cannot write to chat.md
- Concurrent writes safe via rename(2) atomicity on same filesystem — no data corruption possible
- state file update uses same atomic pattern: jq to temp + mv

## Tasks Correction (2026-04-07)

**Reality**: spec-executor.md and external-reviewer.md are agent prompts. Agents execute inline bash commands directly — they NEVER source or call external bash scripts.

**Correction**: Regenerated tasks.md with correct architecture:
- Removed bash script creation tasks (1.3-1.8)
- Replaced with tasks that add FLOC protocol sections directly to agent prompts
- Agents read chat.md with Read tool, write with Write/Bash tool inline
- Tasks count reduced from 46 to 38
- Phase 3 tests rewritten as shell scripts (not bats) with fallback detection
- Task 4.3 fixed to add `version:` field to external-reviewer.md before checking for it

```bash
jq '.taskIndex = 1' .ralph-state.json > /tmp/state.json && mv /tmp/state.json .ralph-state.json
```
This is inline bash, not a script call.
