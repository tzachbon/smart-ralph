**Spec:** `loop-safety-infra`
**Date:** 2026-04-26
**Status:** Research complete

## 1. Best Practices for Automated Git Checkpoints

### 1.1 When to Create the Checkpoint

The checkpoint should be created **immediately before task execution begins** -- that is, at the transition from the `tasks` phase to the `execution` phase. This corresponds to the moment `/ralph-specum:implement` is invoked and the state file transitions to `phase: "execution"` with `taskIndex: 0`.

**Recommended trigger points:**
- **Option A: Stop-watcher hook** -- The stop-watcher already reads state files and controls the execution loop. Adding checkpoint creation here would be elegant since it runs automatically before each loop iteration.
- **Option B: Implement command coordinator** -- The implement.md coordinator prompt is the highest-level entry point and has authoritative knowledge of when execution starts.
- **Option C: Separate PreToolUse hook** -- A dedicated hook that fires when the implement command is first used.

**Recommendation: Option B with Option A as backup.** The implement command is the natural place to create the checkpoint since it controls the execution flow. The stop-watcher can verify the checkpoint exists before proceeding (safety check).

### 1.2 What Gets Checkpointed

```bash
# Recommended checkpoint command
git add -A && git commit -m "checkpoint: before $spec_name execution (task 0/$total_tasks)"
```

Using `git add -A` captures all changes (tracked and deleted). This is important because during spec execution, tasks may create new files, delete old ones, or modify existing ones. We want the checkpoint to represent the full state of the repo before any task execution.

**Important caveat:** `git add -A` will stage untracked files. If the user has other work-in-progress that should not be part of the checkpoint, this could be undesirable. However, for the Smart Ralph context, this is generally the right default because:
1. The plugin is being used in a focused session
2. The spec directory itself contains working files that should be included
3. Rollback is always possible if something goes wrong

### 1.3 Handling Detached HEAD and Non-Main Branches

If the user is on a detached HEAD (e.g., after `git checkout <tag>`), checkpoint creation should fail gracefully:

```bash
BRANCH=$(git branch --show-current 2>/dev/null)
if [ -z "$BRANCH" ]; then
    echo "[ralph-specum] WARNING: Cannot create checkpoint - on detached HEAD" >&2
    echo "[ralph-specum] Proceeding without checkpoint safety net" >&2
    # Still store a "none" checkpoint so we know we skipped
fi
```

### 1.4 Commit Granularity

The checkpoint should be a **single atomic snapshot**, not per-task commits. Reasons:
- Tasks within a spec may leave the repo in a partially broken state between them
- Rolling back to the spec start is cleaner than rolling back multiple task commits
- The checkpoint represents a known-good baseline

## 2. SHA Storage Strategies in State Files

### 2.1 State File Schema

The `.ralph-state.json` already has a rich structure. We need to add checkpoint-related fields:

```json
{
  "phase": "execution",
  "taskIndex": 0,
  "totalTasks": 42,
  "checkpoint": {
    "sha": "a1b2c3d4",
    "timestamp": "2026-04-26T18:30:00Z",
    "branch": "feat/engine-roadmap-epic",
    "message": "checkpoint: before loop-safety-infra execution (task 0/42)"
  },
  ...existing fields...
}
```

**Rationale for nested `checkpoint` object:**
- Keeps checkpoint data grouped logically
- Easy to extend with more fields (e.g., `status: "restored"` after rollback)
- Avoids cluttering the root namespace
- Consistent with how `chat` and `external_unmarks` are organized

### 2.2 SHA Format Considerations

Git SHAs are 40-character hex strings. For storage and display:
- Store full 40-char SHA in the state file (unambiguous)
- Short SHA (7 chars) for display in log messages
- Always verify the SHA exists in the repo after storing: `git cat-file -e "$SHA" 2>/dev/null`

### 2.3 What If Checkpoint Creation Fails?

Three scenarios:

| Scenario | Action | Risk |
|----------|--------|------|
| Git not initialized | Log warning, store `"sha": null` | Medium -- no rollback possible |
| On detached HEAD | Log warning, store `"sha": null` | Medium -- no rollback possible |
| Pre-commit hook fails | Log error, abort execution | Low -- prevents dirty state |
| Permission denied | Log error, abort execution | Low -- prevents corrupted state |

**Critical:** If checkpoint creation fails (except for non-critical cases like detached HEAD), the execution should be **blocked** with a clear error message, rather than proceeding without a safety net.

## 3. Rollback Mechanics

### 3.1 Rollback Strategy: `git reset --soft`

**Recommended rollback command:**

```bash
git reset --soft "$CHECKPOINT_SHA"
```

**Why `--soft`?**
- `--soft`: Resets HEAD and branch pointer to the checkpoint, **keeps all changes staged in the index**. This is ideal because:
  - The working tree is untouched
  - All changes are staged, ready for inspection
  - The user can review what was changed and either commit, amend, or reset further
  - No working tree files are deleted, which avoids potential data loss from conflicting working tree modifications

**Alternatives considered and rejected:**

| Command | Why Rejected |
|---------|-------------|
| `git reset --hard $SHA` | Destroys ALL working tree changes including untracked files. Too destructive for a safety mechanism that may be called mid-execution. |
| `git restore --source=$SHA .` | Only restores tracked files. Leaves new/untracked files. Inconsistent with checkpoint's `git add -A`. |
| `git checkout $SHA` | Detaches HEAD. Changes the branch state, which can confuse the user and other tools. |
| `git reset --mixed $SHA` | Unstages changes. Loses the staging context that `--soft` preserves. |

### 3.2 Rollback Implementation

```bash
#!/bin/bash
# Rollback to the checkpoint SHA stored in a spec's state file
# Usage: rollback-spec <spec_path>

SPEC_PATH="$1"
STATE_FILE="$SPEC_PATH/.ralph-state.json"

if [ ! -f "$STATE_FILE" ]; then
    echo "ERROR: State file not found: $STATE_FILE" >&2
    exit 1
fi

CHECKPOINT_SHA=$(jq -r '.checkpoint.sha // empty' "$STATE_FILE" 2>/dev/null)

if [ -z "$CHECKPOINT_SHA" ]; then
    echo "ERROR: No checkpoint found in state file" >&2
    echo "The spec may not have created a checkpoint, or execution has already completed." >&2
    exit 1
fi

# Verify SHA exists in repo
if ! git cat-file -e "$CHECKPOINT_SHA" 2>/dev/null; then
    echo "ERROR: Checkpoint SHA $CHECKPOINT_SHA not found in repo" >&2
    echo "The checkpoint may have been garbage collected or is from a different repo." >&2
    exit 1
fi

# Perform soft reset
git reset --soft "$CHECKPOINT_SHA"
ROLLBACK_STATUS=$?

if [ $ROLLBACK_STATUS -eq 0 ]; then
    # Mark state as restored
    TMP_FILE=$(mktemp "${STATE_FILE}.tmp.XXXXXX")
    jq '.phase = "post_rollback" | .checkpoint.status = "restored"' "$STATE_FILE" > "$TMP_FILE" && mv "$TMP_FILE" "$STATE_FILE"
    echo "Rolled back to checkpoint $CHECKPOINT_SHA" >&2
    echo "All changes are staged. Review with: git diff --cached" >&2
else
    echo "ERROR: Rollback failed with status $ROLLBACK_STATUS" >&2
    exit 1
fi
```

### 3.3 Post-Rollback State

After rollback, the spec state should reflect the restored condition:

```json
{
  "phase": "post_rollback",
  "checkpoint": {
    "sha": "a1b2c3d4",
    "status": "restored",
    "restoredAt": "2026-04-26T19:00:00Z"
  },
  ...
}
```

This allows the system to:
1. Detect that a rollback occurred (via `phase: "post_rollback"`)
2. Allow the user to decide whether to resume from checkpoint or abandon the spec
3. Provide context about what was rolled back

## 4. Edge Cases

### 4.1 Uncommitted Changes at Checkpoint Time

This is the most critical edge case. When the user invokes `/ralph-specum:implement`, there may be uncommitted changes in the working tree.

**Scenarios:**

1. **Staged changes (git add, not git commit):**
   - `git add -A` will include these in the checkpoint
   - On rollback, `git reset --soft` preserves them as staged changes
   - **Verdict: Safe.** The checkpoint captures the pre-execution state including any staged changes.

2. **Unstaged changes (modified but not staged):**
   - `git add -A` will stage them before commit
   - On rollback, `git reset --soft` keeps them staged
   - **Verdict: Safe.** They are preserved and staged, ready for user review.

3. **Untracked files (new files, not yet added):**
   - `git add -A` will stage them
   - On rollback, they remain as staged new files
   - **Verdict: Safe, but unexpected.** The user should be aware that untracked files are being included in the checkpoint.

4. **Conflicting changes (working tree conflicts):**
   - `git add -A` will fail if there are merge conflicts
   - **Action: Abort checkpoint creation and execution.** Report the conflict clearly.

**Recommended behavior:** Before checkpoint creation, check for conflicts and inform the user:

```bash
# Pre-checkpoint validation
CONFLICTS=$(git ls-files -u 2>/dev/null)
if [ -n "$CONFLICTS" ]; then
    echo "ERROR: Unresolved merge conflicts detected:" >&2
    echo "$CONFLICTS" >&2
    echo "Resolve conflicts or abort Ralph before starting execution." >&2
    exit 1
fi

# Warn about uncommitted changes
STAGED=$(git diff --cached --stat 2>/dev/null)
MODIFIED=$(git diff --stat 2>/dev/null)
UNTRACKED=$(git ls-files --others --exclude-standard 2>/dev/null)

if [ -n "$STAGED" ] || [ -n "$MODIFIED" ] || [ -n "$UNTRACKED" ]; then
    echo "[ralph-specum] WARNING: Uncommitted changes exist. These will be included in the checkpoint." >&2
    [ -n "$STAGED" ] && echo "  Staged changes:" >&2 && echo "$STAGED" >&2
    [ -n "$MODIFIED" ] && echo "  Modified (unstaged):" >&2 && echo "$MODIFIED" >&2
    [ -n "$UNTRACKED" ] && echo "  Untracked files: $(echo $UNTRACKED | wc -w) files" >&2
fi
```

### 4.2 Concurrent Git Operations

If another process (another Claude Code session, a CI system) modifies the repo during execution:

- The checkpoint SHA is still valid in the repo's object store
- On rollback, `git reset --soft` will still work regardless of what HEAD points to
- However, the working tree may have diverged from what was expected
- **Mitigation:** Store the checkpoint timestamp and verify the repo hasn't been tampered with on rollback (optional safety check)

### 4.3 Git Garbage Collection

If `git gc` runs between checkpoint creation and rollback (unlikely in short execution windows, but possible in long-running sessions):

- The checkpoint commit object should survive `git gc` unless it's truly orphaned
- A commit referenced by a state file is not "unreachable" in a normal sense
- However, aggressive `git gc --prune=now` could remove it
- **Mitigation:** Rollback should verify the SHA exists (`git cat-file -e`) before attempting reset

### 4.4 Checkpoint on Already-Behind Branch

If the user has reset their local branch back several commits and then runs Ralph:

- The checkpoint will be created on top of the current HEAD (which is behind where it might have been)
- On rollback, `git reset --soft` moves HEAD back to the checkpoint
- This is correct behavior -- the checkpoint represents the last-known state before execution

### 4.5 Submodule State

If the repo contains submodules:

- `git add -A` does not automatically stage submodule updates
- The checkpoint may not capture submodule state changes
- **Recommendation:** Document this limitation. For most Ralph use cases (plugin development), submodules are not used.

### 4.6 Stash Operations

If the user has stashed changes:

- Stashes are stored as separate commit objects and are not affected by `git add -A` or `git reset`
- Stashed changes survive checkpoint creation and rollback
- **Verdict: Safe.** Stashes are independent of the main branch state.

## 5. How Other Tools Handle This

### 5.1 GitHub Actions

GitHub Actions does not create checkpoints by default. Each job runs on a fresh checkout:

```yaml
- uses: actions/checkout@v4
  with:
    persist-credentials: false
```

For stateful jobs, they use `actions/cache` or `actions/upload-artifact` to persist state between jobs. When they need to rollback, they typically:
- Use `git checkout <previous-sha>` if they've moved HEAD
- Use artifact restore instead of git rollback

**Relevance to Ralph:** Unlike CI where jobs are isolated, Ralph runs interactively in the user's working tree, so we need the checkpoint approach (not checkout-based isolation).

### 5.2 Argo Workflows

Argo Workflows supports checkpointing via:
- **Artifact storage:** Saves the entire workspace to S3/GCS between workflow steps
- **Exit hooks:** Runs cleanup/rollback actions on failure via `onSuccess`/`onFailure`

Example pattern:
```yaml
- name: checkpoint
  steps:
    - - name: run-task
        template: task-template
      - name: rollback-on-failure
        template: rollback-template
        onExit: true
```

**Relevance:** The Argo pattern of "run cleanup on exit" is relevant. Ralph could implement a similar pattern using a PostToolUse or Stop hook that detects execution failure and triggers rollback.

### 5.3 CI/CD Systems

**Jenkins:** Uses `sh` steps with `git stash`/`git stash pop` for temporary state:
```groovy
sh 'git stash push -m "checkpoint before build"'
try {
    // build steps
} finally {
    sh 'git stash pop'  // or: sh 'git checkout HEAD -- .'
}
```

**GitLab CI:** Similar -- each pipeline runs on a fresh checkout via `git clone`.

**Bazel (Build System):** Uses `bazel run` which creates a sandboxed environment. Rollback is implicit because each run is independent.

### 5.4 Deployment Tools

**Capistrano (Ruby):** Creates timestamped releases:
```bash
ln -s /path/to/releases/20260426183000 /path/to/current
# On rollback:
ln -sfn /path/to/releases/20260425120000 /path/to/current
```
This is a symlink-based approach that doesn't modify existing releases.

**Flyway (DB migrations):** Stores migration checksums in a schema history table. Rollback runs reverse migrations.

**Relevance:** The symlink approach (Capistrano) is conceptually useful -- it means the old release is never modified. However, for Ralph's in-place editing model, we can't use symlinks. The checkpoint+soft-reset approach is the closest analog.

### 5.5 Developer Tools

**`git stash` + `git stash pop`:**
```bash
git stash push -m "ralph-checkpoint"
# ... execution ...
git stash pop  # or: git stash drop  (to discard)
```
- Simpler than commit-based checkpoint
- Loses the ability to see what changed (no commit diff)
- Fails if working tree has uncommitted changes that conflict with stash
- **Not recommended for Ralph** because we want diff visibility on rollback.

**`direnv` / `envdir`:** Not relevant -- these are environment isolation, not filesystem state.

### 5.6 AI Coding Assistants

**Codex (OpenAI):** Runs in a sandboxed container. No git checkpoint needed because the environment is ephemeral.

**Claude Code (Anthropic):** Has built-in "worktree" support (`git worktree add`). When Claude Code opens a worktree, it operates in an isolated branch:
```bash
git worktree add .claude/worktrees/<id> HEAD
# All work happens in the worktree branch
# Cleanup is just: git worktree remove <worktree>
```
This is the cleanest isolation model -- it avoids the need for checkpoints entirely because changes are on a separate branch.

**Relevance:** The worktree approach is the ideal solution. However, it requires user setup and is not automatic. The checkpoint approach is a fallback for when worktrees are not available.

## 6. Recommendations for Smart Ralph Implementation

### 6.1 Architecture Decision

**Primary approach: Git commit checkpoint (as described above)**
**Secondary fallback: Worktree mode (if available)**

The commit-based checkpoint is:
- Automatic (no user setup)
- Works with any git repo
- Provides diff visibility on rollback
- Compatible with the existing stop-watcher architecture

Worktree mode should be explored as a separate enhancement (see the existing `superpowers:using-git-worktrees` skill).

### 6.2 Implementation Plan

#### 6.2.1 Checkpoint Script (`hooks/scripts/checkpoint-create.sh`)

```bash
#!/usr/bin/env bash
# Creates a git checkpoint before spec execution.
# Reads state from stdin (hook input format).
# Outputs SHA of created commit.

set -euo pipefail

INPUT=$(cat)
CWD=$(echo "$INPUT" | jq -r '.cwd // empty' 2>/dev/null || true)
if [ -z "$CWD" ]; then
    exit 0
fi

source "$(dirname "${BASH_SOURCE[0]}")/path-resolver.sh"
export RALPH_CWD="$CWD"

SPEC_RELATIVE_PATH=$(ralph_resolve_current 2>/dev/null)
if [ -z "$SPEC_RELATIVE_PATH" ]; then
    exit 0
fi

SPEC_PATH="$CWD/$SPEC_RELATIVE_PATH"
STATE_FILE="$SPEC_PATH/.ralph-state.json"

# Guard: only create checkpoint at execution start
PHASE=$(jq -r '.phase // empty' "$STATE_FILE" 2>/dev/null || true)
TASK_INDEX=$(jq -r '.taskIndex // -1' "$STATE_FILE" 2>/dev/null || true)
EXISTING_SHA=$(jq -r '.checkpoint.sha // empty' "$STATE_FILE" 2>/dev/null || true)

# Skip if checkpoint already exists (idempotent)
if [ -n "$EXISTING_SHA" ]; then
    echo "checkpoint:$EXISTING_SHA"
    exit 0
fi

# Skip if not at execution start
if [ "$PHASE" != "execution" ] || [ "$TASK_INDEX" != "0" ]; then
    exit 0
fi

SPEC_NAME=$(basename "$SPEC_PATH")

# Guard: must be in a git repo
if ! git rev-parse --is-inside-work-tree >/dev/null 2>&1; then
    echo "[ralph-specum] WARNING: Not in a git repo. Skipping checkpoint." >&2
    # Store null checkpoint
    jq '.checkpoint = {"sha": null, "reason": "not_a_git_repo"}' "$STATE_FILE" > "${STATE_FILE}.tmp" && mv "${STATE_FILE}.tmp" "$STATE_FILE"
    echo "checkpoint:null"
    exit 0
fi

# Guard: not on detached HEAD
BRANCH=$(git branch --show-current 2>/dev/null || true)
if [ -z "$BRANCH" ]; then
    echo "[ralph-specum] WARNING: On detached HEAD. Skipping checkpoint." >&2
    jq '.checkpoint = {"sha": null, "reason": "detached_head"}' "$STATE_FILE" > "${STATE_FILE}.tmp" && mv "${STATE_FILE}.tmp" "$STATE_FILE"
    echo "checkpoint:null"
    exit 0
fi

# Warn about uncommitted changes
STAGED=$(git diff --cached --stat 2>/dev/null | head -20)
MODIFIED=$(git diff --stat 2>/dev/null | head -20)
UNTRACKED_COUNT=$(git ls-files --others --exclude-standard 2>/dev/null | wc -l)

if [ -n "$STAGED" ] || [ -n "$MODIFIED" ] || [ "$UNTRACKED_COUNT" -gt 0 ]; then
    echo "[ralph-specum] Note: ${UNTRACKED_COUNT} untracked, modified and/or staged files will be included in the checkpoint." >&2
fi

# Create checkpoint commit
CHECKPOINT_MSG="checkpoint: before ${SPEC_NAME} execution (task 0/$TASK_INDEX)"
if git add -A && git commit -m "$CHECKPOINT_MSG" --allow-empty; then
    CHECKPOINT_SHA=$(git rev-parse HEAD)
    TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ")

    # Update state file with checkpoint info
    TMP_FILE=$(mktemp "${STATE_FILE}.tmp.XXXXXX")
    jq --arg sha "$CHECKPOINT_SHA" \
       --arg ts "$TIMESTAMP" \
       --arg branch "$BRANCH" \
       --arg msg "$CHECKPOINT_MSG" \
       '.checkpoint = {"sha": $sha, "timestamp": $ts, "branch": $branch, "message": $msg}' \
       "$STATE_FILE" > "$TMP_FILE" && mv "$TMP_FILE" "$STATE_FILE"

    echo "checkpoint:$CHECKPOINT_SHA"
else
    echo "[ralph-specum] ERROR: Checkpoint commit failed. Execution may proceed without safety net." >&2
    echo "checkpoint:failure"
    exit 1
fi
```

#### 6.2.2 Rollback Script (`hooks/scripts/checkpoint-rollback.sh`)

```bash
#!/usr/bin/env bash
# Rolls back to the checkpoint SHA stored in a spec's state file.

set -euo pipefail

INPUT=$(cat)
CWD=$(echo "$INPUT" | jq -r '.cwd // empty' 2>/dev/null || true)
if [ -z "$CWD" ]; then
    exit 1
fi

source "$(dirname "${BASH_SOURCE[0]}")/path-resolver.sh"
export RALPH_CWD="$CWD"

SPEC_RELATIVE_PATH=$(ralph_resolve_current 2>/dev/null)
if [ -z "$SPEC_RELATIVE_PATH" ]; then
    echo "ERROR: No active spec resolved" >&2
    exit 1
fi

STATE_FILE="$CWD/$SPEC_RELATIVE_PATH/.ralph-state.json"
if [ ! -f "$STATE_FILE" ]; then
    echo "ERROR: State file not found: $STATE_FILE" >&2
    exit 1
fi

CHECKPOINT_SHA=$(jq -r '.checkpoint.sha // empty' "$STATE_FILE" 2>/dev/null)
if [ -z "$CHECKPOINT_SHA" ]; then
    echo "ERROR: No checkpoint SHA found in state file" >&2
    echo "Available fields: $(jq 'keys' "$STATE_FILE")" >&2
    exit 1
fi

# Verify SHA exists
if ! git cat-file -e "$CHECKPOINT_SHA" 2>/dev/null; then
    echo "ERROR: Checkpoint SHA $CHECKPOINT_SHA not found in git object store" >&2
    echo "The checkpoint may have been pruned." >&2
    exit 1
fi

# Perform soft reset
echo "[ralph-specum] Rolling back to checkpoint $CHECKPOINT_SHA..." >&2
if git reset --soft "$CHECKPOINT_SHA"; then
    TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
    TMP_FILE=$(mktemp "${STATE_FILE}.tmp.XXXXXX")
    jq --arg ts "$TIMESTAMP" \
       '.phase = "post_rollback" | .checkpoint.status = "restored" | .checkpoint.restoredAt = $ts | .taskIndex = 0' \
       "$STATE_FILE" > "$TMP_FILE" && mv "$TMP_FILE" "$STATE_FILE"

    echo "[ralph-specum] Rollback complete. Changes are staged." >&2
    echo "[ralph-specum] Review with: git diff --cached" >&2
    echo "[ralph-specum] Restore checkpoint commit with: git reset --soft $CHECKPOINT_SHA^" >&2
    exit 0
else
    echo "[ralph-specum] ERROR: Rollback failed" >&2
    exit 1
fi
```

#### 6.2.3 Hook Integration

Add checkpoint creation to the Stop hook chain. The stop-watcher already runs before each loop iteration. We add a checkpoint check at the start:

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/hooks/scripts/checkpoint-create.sh"
          },
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/hooks/scripts/stop-watcher.sh"
          }
        ]
      }
    ]
  }
}
```

The `checkpoint-create.sh` runs first (hooks in array execute in order). It creates the checkpoint if one doesn't exist. Then `stop-watcher.sh` proceeds with its normal loop control logic.

#### 6.2.4 Rollback Trigger

Rollback should be a separate command rather than automatic, triggered by the user:

```
/ralph-specum:rollback    # Rollback to checkpoint of current spec
/ralph-specum:rollback --spec <name>    # Rollback a specific spec
```

**Why not automatic?** Automatic rollback on failure could mask real bugs and confuse the user. The user should explicitly invoke rollback after seeing that execution went wrong.

### 6.4 Testing Strategy

1. **Happy path:** Create a spec, start execution, verify checkpoint commit exists and SHA is stored in state file
2. **Rollback happy path:** Rollback after partial execution, verify `git reset --soft` worked, verify state file updated
3. **No-git-repo:** Run Ralph outside a git repo, verify graceful handling
4. **Detached HEAD:** Run Ralph on detached HEAD, verify graceful handling
5. **Existing checkpoint:** Verify idempotency (second invocation doesn't create duplicate)
6. **Pre-existing changes:** Run Ralph with staged/unstaged/untracked changes, verify they are included in checkpoint
7. **Failed commit:** If pre-commit hook fails, verify execution is blocked

### 6.5 Known Limitations

1. **No checkpoint for parallel specs:** If two specs run in parallel (same repo), checkpoints may interfere. This is acceptable because Ralph typically runs one spec at a time.

2. **Checkpoint commit visible in history:** The checkpoint commits will appear in `git log`. This is generally desirable (audibility) but could clutter history for large specs. Consider `--no-verify` on checkpoint commits to skip pre-commit hooks (security note: this is safe because we trust the checkpoint content).

3. **Submodule state not captured:** `git add -A` doesn't stage submodule commits. If a spec modifies submodule references, the checkpoint may not capture this accurately.

4. **Large repos:** `git add -A` can be slow in very large repos (100k+ files). The `--all` flag with a specific path (`. -include "specs/*"`) could optimize this, but risks missing changes outside the specs directory.

### 6.6 Future Enhancements

1. **Per-task checkpoints:** For very long specs (100+ tasks), create checkpoints every N tasks to limit rollback scope.

2. **Worktree mode:** As mentioned in Section 5.6, using git worktrees provides full isolation. This should be explored as a separate enhancement, perhaps gated by a `--worktree` flag on the implement command.

3. **Checkpoint diff display:** After rollback, show a diff summary of what changed between checkpoint and rollback point.

4. **Checkpoint retention policy:** Automatically prune old checkpoint commits after spec completion to keep history clean.

## Summary of Recommendations

| Decision | Recommendation | Rationale |
|----------|---------------|-----------|
| Checkpoint timing | Execution start (tasks phase -> execution phase) | Cleanest boundary, matches spec lifecycle |
| Checkpoint command | `git add -A && git commit` | Captures all changes including new files |
| Rollback command | `git reset --soft $SHA` | Preserves changes staged, no data loss |
| SHA storage | Nested `checkpoint` object in state file | Logical grouping, extensible |
| Failure handling | Abort execution if checkpoint fails | Safety first, except for non-critical cases |
| Rollback trigger | Explicit user command (`/ralph-specum:rollback`) | User decides when to rollback |
| Hook integration | Prepend checkpoint script to Stop hook chain | Automatic, no extra user action |
| Idempotency | Skip if checkpoint SHA already exists | Prevents duplicate checkpoints |
