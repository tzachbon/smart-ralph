# Research: Per-Task Metrics (JSONL) & CI Snapshot Tracking

## 1. JSONL Metrics Patterns

### 1.1 Why JSONL over JSON

JSONL (newline-delimited JSON) is the standard for append-only event logs. Benefits:

- **Append-safe**: Each line is a self-contained document. No need to rewrite the file or acquire exclusive locks beyond what flock already provides.
- **Stream-processable**: Can be consumed line-by-line by tools (jq, awk, grep) without loading the full file.
- **Crash-resilient**: A crash mid-write corrupts only one line, not the entire file. A JSON array would be unrecoverable.
- **Scalable**: Growing file does not change read/write characteristics for new lines.

**Frameworks that use this pattern:**
- **GitHub Actions**: Each job step writes to `$GITHUB_STEP_SUMMARY` and `$GITHUB_OUTPUT` as structured text. The runner itself logs in JSONL-like format.
- **LangGraph/Prefect**: Both use JSONL for run tracing (`runs.jsonl`), each entry a complete execution snapshot.
- **OpenTelemetry**: Uses JSONL for trace exports when file exporters are configured.
- **pytest-xdist + JSONL**: Custom pytest plugins write per-test metrics as JSONL lines to enable post-hoc analysis.

### 1.2 Schema Design

```jsonc
{
  // ── Identity ──
  "eventId": "uuid-v4",              // Unique ID for dedup/idempotency
  "spec": "spec-name",               // Spec being executed
  "specPath": "/absolute/path/to/spec",
  "timestamp": "2026-04-26T14:30:00Z", // ISO 8601 with timezone
  "globalIteration": 42,             // From .ralph-state.json
  "taskIndex": 7,                    // 0-based, from .ralph-state.json
  "taskIteration": 2,                // Retry count for this task
  "phase": "execution",              // From .ralph-state.json

  // ── Task metadata ──
  "taskTitle": "Implement user auth", // From tasks.md (first line)
  "taskType": "implementation",      // See §1.3 task type classification
  "taskId": "8.3",                   // Section.task format from tasks.md

  // ── Timing ──
  "startedAt": "2026-04-26T14:30:00Z",
  "completedAt": "2026-04-26T14:32:15Z",
  "wallTimeMs": 135000,              // Total elapsed (ms)
  "verifyTimeMs": 5200,              // Time spent on verify command (ms)

  // ── Outcome ──
  "status": "pass",                  // pass | fail | timeout | cancelled | ambiguous
  "commit": "a1b2c3d",               // 7-char hash, null if no commit
  "verifyResult": "all tests passed (12/12)", // Human-readable verify output
  "verifyExitCode": 0,               // Exit code of verify command
  "verifiedBy": "spec-executor",     // Who ran verification

  // ── Retries ──
  "retries": 1,                      // Number of retry attempts within this task

  // ── Errors (present only on failure) ──
  "error": null,                     // One-line error summary, null on pass
  "errorDetail": null,               // Multi-line / structured error context

  // ── CI snapshot (see §5) ──
  "ciSnapshotBefore": null,          // {lint: "pass", test: "pass", ...} or null
  "ciSnapshotAfter": null,           // Same schema, captured post-task
  "ciDrift": false,                  // Whether CI state changed

  // ── Agent context ──
  "agent": "spec-executor",          // Which agent executed
  "toolsUsed": ["Read", "Edit", "Bash", "Write"], // Claude Code tools invoked

  // ── Flags ──
  "isCheckpoint": false,             // Quality checkpoint task?
  "isVerifyTask": false,             // [VERIFY] task?
  "recoveryMode": false,             // From state
  "quickMode": false                 // From state
}
```

**Design decisions:**

| Decision | Rationale |
|----------|-----------|
| `wallTimeMs` in ms, not seconds | Sub-second precision matters for fast tasks (e.g., lint) |
| Separate `verifyTimeMs` | Isolates CI/verify cost from implementation cost |
| `retries` as count, not array | Simpler aggregation; detailed retry errors go in `.progress.md` |
| `ciSnapshotBefore` / `ciSnapshotAfter` | Enables drift detection: did this task break CI? |
| `toolsUsed` array | Helps identify which agents/agents-patterns are most expensive |
| `errorDetail` nullable | Most entries have no error; keep the happy path minimal |

### 1.3 Task Type Classification

Classify tasks for aggregation:

| Type | Detection | Examples |
|------|-----------|----------|
| `implementation` | Default | Regular [ ] tasks |
| `test_red` | [RED] tag | Write failing test |
| `test_green` | [GREEN] tag | Make test pass |
| `test_yellow` | [YELLOW] tag | Refactor with tests |
| `verification` | [VERIFY] tag | Quality checkpoint |
| `commit` | PR creation, state updates | Coordinator-only work |
| `repair` | Recovery-mode fix tasks | Auto-generated fix tasks |
| `modification` | SPLIT_TASK, ADD_PREREQUISITE | Task restructuring |

Classification logic for bash:

```bash
classify_task_type() {
    local task_block="$1"
    if echo "$task_block" | grep -q '\[VERIFY\]'; then
        echo "verification"
    elif echo "$task_block" | grep -q '\[RED\]'; then
        echo "test_red"
    elif echo "$task_block" | grep -q '\[GREEN\]'; then
        echo "test_green"
    elif echo "$task_block" | grep -q '\[YELLOW\]'; then
        echo "test_yellow"
    else
        echo "implementation"
    fi
}
```

### 1.4 Append-Only Write Strategy

**File locking**: Use flock for atomic append, same pattern as chat.md in the existing codebase:

```bash
METRICS_FILE="${SPEC_PATH}/.metrics.jsonl"
(
    exec 201>"${METRICS_FILE}.lock"
    flock -x 201

    # Atomic append: read existing line count, append new entry
    cat >> "$METRICS_FILE" << 'METRICSEOF'
{"eventId":"uuid","spec":"...","timestamp":"...","taskIndex":0,"status":"pass","commit":"a1b2c3d",...}
METRICSEOF
) 201>"${METRICS_FILE}.lock"
```

**When to write entries:**

| Trigger | Entry type |
|---------|-----------|
| Task starts | `metric_start` — timestamp, task metadata only |
| Task completes (pass/fail) | `metric_complete` — full entry with timing, outcome |
| Verification fails | `metric_verify_fail` — includes verify output, exit code |
| Task timeout | `metric_timeout` — elapsed time, last tool used |
| Spec completes | `metric_spec_complete` — summary aggregations |

**Lifecycle**: The `.metrics.jsonl` file is created on first task execution and persists after spec completion (like `.progress.md`). It is NOT deleted during cleanup.

**Read pattern**: Consumers read from the end (recent entries) using `tail`, or scan entire file for aggregations using `jq -s` (slurp).

## 3. CI Command Discovery

### 3.1 Files to Scan (in priority order)

| File | Commands Found | Priority |
|------|---------------|----------|
| `package.json` | `"scripts"` object keys | **P0** — npm/yarn/pnpm projects |
| `Makefile` / `GNUmakefile` / `makefile` | Target names (non-.PHONY) | **P0** — C/Go/Python/infra projects |
| `.github/workflows/*.yml` | `run:` commands in jobs | **P1** — GitHub Actions CI |
| `Jenkinsfile` / `Jenkinsfile.*` | `stage`/`step` blocks | **P1** — Jenkins CI |
| `ci.yml` / `ci.yaml` / `azure-pipelines.yml` | `script:`/`task:` blocks | **P2** — Other CI configs |
| `.gitlab-ci.yml` | Top-level keys | **P2** — GitLab CI |
| `pyproject.toml` | `[tool.pytest]`, `[tool.ruff]` | **P2** — Python projects |
| `Cargo.toml` | `"scripts"` or build system | **P2** — Rust projects |
| `Justfile` / `Rakefile` | Target/command definitions | **P3** — Alternative build systems |

### 3.2 Discovery Logic

```bash
discover_ci_commands() {
    local root="$1"  # project root, default: repo root
    local commands=()
    local types=()

    # P0: package.json scripts
    if [ -f "$root/package.json" ]; then
        while IFS= read -r line; do
            local name=$(echo "$line" | cut -d: -f1 | tr -d ' "')
            local cmd=$(echo "$line" | cut -d: -f2- | sed 's/^[[:space:]]*//' | sed 's/,$//')
            commands+=("$cmd")
            types+=("npm:$name")
        done < <(python3 -c "
import json, sys
with open('$root/package.json') as f:
    scripts = json.load(f).get('scripts', {})
    for k, v in scripts.items():
        print(f'{k}: {v}')
" 2>/dev/null)
    fi

    # P0: Makefile targets
    if [ -f "$root/Makefile" ] || [ -f "$root/GNUmakefile" ]; then
        while IFS= read -r line; do
            local target=$(echo "$line" | grep -oP '^[a-zA-Z0-9_-]+(?=\s*:)' | head -1)
            if [ -n "$target" ] && [ "$target" != ".PHONY" ]; then
                commands+=("make $target")
                types+=("make:$target")
            fi
        done < <(grep -E '^[a-zA-Z0-9_-]+\s*:' "$root/Makefile" "$root/GNUmakefile" 2>/dev/null)
    fi

    # P1: GitHub Actions workflows
    if [ -d "$root/.github/workflows" ]; then
        for wf in "$root/.github/workflows/"*.yml "$root/.github/workflows/"*.yaml; do
            [ -f "$wf" ] || continue
            # Extract run commands and stage names
            while IFS= read -r line; do
                commands+=("github-actions:$(basename "$wf"): $line")
                types+=("github-actions:$(basename "$wf")")
            done < <(grep -oP '(?<=run: )[^\n]+' "$wf" 2>/dev/null)
        done
    fi

    # Output: tab-separated command\ttype
    for i in "${!commands[@]}"; do
        printf '%s\t%s\n' "${commands[$i]}" "${types[$i]}"
    done
}
```

### 3.3 Command Classification

After discovery, classify each command by its dominant pattern:

```bash
classify_command() {
    local cmd="$1"
    local cmd_lower=$(echo "$cmd" | tr '[:upper:]' '[:lower:]')

    # Test
    if echo "$cmd_lower" | grep -qE '(test|spec|assert|pytest|jest|mocha|tap|cypress|playwright)'; then
        echo "test"
        return
    fi

    # Lint
    if echo "$cmd_lower" | grep -qE '(lint|eslint|prettier|black|flake8|ruff|mypy|pylint|clippy)'; then
        echo "lint"
        return
    fi

    # Build
    if echo "$cmd_lower" | grep -qE '(build|compile|tsc|webpack|rollup|esbuild|cargo build|go build|npm run build)'; then
        echo "build"
        return
    fi

    # Deploy
    if echo "$cmd_lower" | grep -qE '(deploy|publish|release|push|tag|ship|release-it|semantic-release)'; then
        echo "deploy"
        return
    fi

    # Type check
    if echo "$cmd_lower" | grep -qE '(typecheck|type-check|tsc --noEmit|pyright|mypy --strict)'; then
        echo "typecheck"
        return
    fi

    # Format
    if echo "$cmd_lower" | grep -qE '(format|prettier|black --check|gofmt)'; then
        echo "format"
        return
    fi

    # Security
    if echo "$cmd_lower" | grep -qE '(security|sca|sarif|dependabot|audit|snyk)'; then
        echo "security"
        return
    fi

    # Fallback: classify by script name prefix
    if echo "$cmd_lower" | grep -qE '^(check|validate|verify|ci|quality|safety)'; then
        echo "check"
    else
        echo "other"
    fi
}
```

**Classification matrix:**

| Category | Example commands | CI gate importance |
|----------|-----------------|-------------------|
| `test` | `npm test`, `pytest`, `make test` | **MUST PASS** — hard gate |
| `lint` | `npm run lint`, `ruff check` | **SHOULD PASS** — soft gate (CI warning) |
| `build` | `npm run build`, `make build` | **MUST PASS** — hard gate |
| `typecheck` | `tsc --noEmit`, `pyright` | **SHOULD PASS** — soft gate |
| `deploy` | `npm publish`, `gh release` | **N/A** — not gated (manual action) |
| `format` | `prettier --check` | **WILL NOT GATE** — informational |
| `security` | `npm audit`, `snyk test` | **SHOULD PASS** — soft gate |

### 3.4 Edge Cases

| Scenario | Handling |
|----------|----------|
| **Command requires credentials** (e.g., `npm publish --access public`) | Skip from auto-gating; record in discovered list but do NOT auto-run |
| **Command requires external services** (e.g., DB, API) | Record as `requires: external`. Run only if user opts in or in CI environment |
| **Command takes > 5 minutes** (e.g., full regression suite) | Record but do NOT run per-task. Mark as `long_running: true`. Offer as opt-in sweep |
| **Command has side effects** (e.g., `make deploy`, `npm publish`) | Record but do NOT auto-execute. Flag as `side_effect: true` |
| **Command requires a virtual environment** (e.g., `source venv/bin/activate && pytest`) | Extract as a wrapper: `setup_cmd: "source venv/bin/activate"` |
| **Command has conditional flags** (e.g., `npm test -- --coverage`) | Store raw command; run without extra flags by default |
| **No CI config found** | Set `ciCommands: []` in state. Allow user to manually specify via `ciCommands` field |
| **Multiple matching files** (e.g., both `Makefile` and `package.json`) | Merge all discovered commands. No conflicts; each is independent |

## 4. CI Snapshot: Capture, Store, Pass/Fail Criteria

### 4.1 What Is a CI Snapshot

A CI snapshot is a point-in-time capture of each CI command's result, stored as structured data in `.ralph-state.json` under a `ciSnapshot` field:

```jsonc
{
  "ciSnapshot": {
    "capturedAt": "2026-04-26T14:25:00Z",
    "capturedBy": "coordinator",
    "commands": {
      "lint": {
        "command": "ruff check src/",
        "status": "pass",
        "exitCode": 0,
        "durationMs": 3200,
        "outputSummary": "0 issues found"
      },
      "test": {
        "command": "pytest tests/ --cov=src -q",
        "status": "pass",
        "exitCode": 0,
        "durationMs": 12400,
        "outputSummary": "142 passed, 3 skipped"
      },
      "build": {
        "command": "tsc --noEmit",
        "status": "pass",
        "exitCode": 0,
        "durationMs": 8100,
        "outputSummary": "Found 0 errors"
      },
      "typecheck": {
        "command": "pyright src/",
        "status": "pass",
        "exitCode": 0,
        "durationMs": 5600,
        "outputSummary": "0 errors, 0 warnings"
      }
    }
  }
}
```

### 4.2 When to Capture

| Capture Point | Timing | Purpose |
|---------------|--------|---------|
| **Pre-execution** | After state init, before first task | Baseline CI state at spec start |
| **Post-task** | After each task's verify passes | Detect task-induced CI regression |
| **On failure** | When a task's verify fails | Capture what CI looked like before the break |
| **Spec complete** | After all tasks pass | Final CI state, recorded in `.progress.md` |

**Implementation trigger**: The stop-watcher hook captures pre-execution snapshot during `implement.md` Step 3 (state initialization). Post-task snapshots are captured by the spec-executor's `<output_protocol>` block (or by the coordinator after TASK_COMPLETE).

### 4.3 Pass/Fail Criteria

| Command category | Pass condition | Fail condition |
|-----------------|---------------|----------------|
| `test` | Exit code 0 AND at least one test ran | Exit code != 0 OR "0 tests ran" in output |
| `lint` | Exit code 0 | Exit code != 0 |
| `build` | Exit code 0 | Exit code != 0 |
| `typecheck` | Exit code 0 | Exit code != 0 |
| `format` | Exit code 0 | Exit code != 0 (formatting violations found) |
| `security` | Exit code 0 AND severity < "high" | Exit code != 0 OR any "high" severity finding |

**Important**: A "pass" on lint/typecheck does NOT block task advancement if the command was NOT in the pre-execution snapshot (i.e., it was newly discovered). Only commands that passed pre-execution AND now fail constitute a drift.

### 4.4 Drift Detection

```bash
detect_ci_drift() {
    local spec_path="$1"
    local state_file="$spec_path/.ralph-state.json"

    # Read pre-execution snapshot
    local snapshot=$(jq '.ciSnapshot.commands // {}' "$state_file" 2>/dev/null)
    [ -z "$snapshot" ] && return 0  # No snapshot to compare against

    local has_drift=false

    for category in lint test build typecheck; do
        local cmd=$(echo "$snapshot" | jq -r --arg cat "$category" '.[$cat].command // empty' 2>/dev/null)
        [ -z "$cmd" ] && continue

        # Run command
        local exit_code=0
        local output
        output=$($cmd 2>&1) || exit_code=$?

        # Compare
        local pre_status=$(echo "$snapshot" | jq -r --arg cat "$category" '.[$cat].status' 2>/dev/null)

        if [ "$pre_status" = "pass" ] && [ "$exit_code" -ne 0 ]; then
            echo "CI_DRIFT: $category passed at snapshot start, now fails (exit $exit_code)"
            has_drift=true
        elif [ "$pre_status" = "fail" ] && [ "$exit_code" -eq 0 ]; then
            echo "CI_RECOVERED: $category was failing at snapshot start, now passes"
        fi
    done

    [ "$has_drift" = true ] && echo "CI_DRIFT_DETECTED" || echo "CI_DRIFT_NONE"
}
```

### 4.5 CI Snapshot Lifecycle

```
[State Init] ──capture──> [Pre-snapshot stored in .ralph-state.json]
      │
      ▼
[Task 1] ──verify passes──> [Post-task CI re-run? (optional, see §4.6)]
      │
      ▼
[Task N] ──verify passes──> [Final CI check at spec completion]
      │
      ▼
[Spec Complete] ──> [CI snapshot recorded in .progress.md, state deleted]
```

### 4.6 Performance Trade-offs

Running CI commands after every task adds overhead. Two strategies:

| Strategy | When to run | Overhead | Coverage |
|----------|-------------|----------|----------|
| **Full post-task CI** | Every task passes verify | High (seconds to minutes per task) | Maximum |
| **Selective post-task CI** | Only tasks modifying source files (not tests/docs) | Medium | Good |
| **Spec-complete CI** | Only after all tasks pass | Low (once per spec) | Moderate |

**Recommendation**: Use selective strategy. After each task:
1. Check `git diff --name-only HEAD` to see what files changed
2. If only test files changed → skip full CI, trust task verify
3. If source files changed → run at minimum `lint` + `typecheck` (fast commands)
4. Reserve `test` and `build` for spec completion

Implementation:

```bash
should_run_ci_after_task() {
    local spec_path="$1"
    local changed_files=$(git diff --name-only HEAD 2>/dev/null || true)

    # If only test/spec files changed, skip CI (task verify is sufficient)
    if echo "$changed_files" | grep -qE '^(tests?/|__tests__/|\.test\.|\.spec\.)'; then
        # Check if ANY non-test files changed
        local non_test=$(echo "$changed_files" | grep -vE '(tests?/|__tests__/|\.test\.|\.spec\.)' || true)
        if [ -z "$non_test" ]; then
            echo "SKIP"  # Only test files changed
            return
        fi
    fi

    # Source files changed — run fast CI (lint + typecheck)
    echo "RUN_FAST"
}
```

## 5. Integration Points in Stop-Watcher

### 5.1 Where Metrics Writing Happens

The metrics entry should be written by the **coordinator** (in `implement.md`), not by the stop-watcher. The stop-watcher is read-only regarding state.

**Coordinator writes metrics on:**
- Task start: minimal entry (identity + timestamp)
- Task complete: full entry (outcome + timing + verify result)
- Spec complete: summary entry

**Stop-watcher reads metrics on:**
- Drift detection: check if `.metrics.jsonl` has recent fail entries
- Circuit breaker: count consecutive failures from metrics

### 5.2 Integration with Existing State

```jsonc
// Additions to .ralph-state.json
{
  // ... existing fields ...
  "ciCommands": [],              // Discovered CI commands (string[])
  "ciSnapshot": {                // CI snapshot data (object, see §4.1)
    "capturedAt": "...",
    "commands": {}
  },
  "metricsFile": ".metrics.jsonl", // Path for consistency
  "circuitBreaker": {            // Circuit breaker state (from spec goal)
    "consecutiveFailures": 0,
    "lastFailureAt": null,
    "maxConsecutiveFailures": 5,
    "maxAgeHours": 48
  }
}
```

### 5.3 Metrics Entry Creation (Bash Pattern)

```bash
write_metric() {
    local spec_path="$1"
    local status="$2"
    local task_index="$3"
    local task_iteration="$4"
    local task_title="$5"
    local commit="$6"
    local verify_result="$7"
    local verify_exit="$8"
    local started_epoch="$9"
    local agent="${10}"
    local tools_used="${11}"

    local metrics_file="${spec_path}/.metrics.jsonl"
    local now_epoch=$(date +%s)
    local wall_time_ms=$(( (now_epoch - started_epoch) * 1000 ))
    local now_iso=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
    local event_id=$(cat /proc/sys/kernel/random/uuid 2>/dev/null || echo "evt-${task_index}-${task_iteration}-${now_epoch}")

    # Classify task type
    local task_type="implementation"
    local task_block=$(grep -A20 "^\- \[[ x]\].*\$((task_index + 1))\." "${spec_path}/tasks.md" 2>/dev/null || echo "")
    task_type=$(classify_task_type "$task_block")

    jq -n \
      --arg event_id "$event_id" \
      --arg spec "$(basename "$spec_path")" \
      --arg spec_path "$spec_path" \
      --arg timestamp "$now_iso" \

      --argjson task_index "$task_index" \
      --argjson task_iteration "$task_iteration" \
      --arg phase "execution" \
      --arg task_title "$task_title" \
      --arg task_type "$task_type" \

      --arg commit "$commit" \
      --arg verify_result "$verify_result" \

      }' >> "$metrics_file"
}
```

### 5.4 CI Command Discovery in implement.md

Add to `implement.md` Step 3 (Initialize Execution State), before writing `.ralph-state.json`:

```bash
# CI Command Discovery
discover_and_store_ci_commands() {
    local spec_path="$1"
    local repo_root="$CWD"

    # Determine project root (may differ from CWD for monorepos)
    local project_root="$repo_root"
    if [ -f "$CWD/package.json" ]; then
        project_root=$(cd "$CWD" && pwd)
    fi

    local discovered_commands=()
    local classified_commands="{}"

    # Discover from package.json
    if [ -f "$project_root/package.json" ]; then
        while IFS=$'\t' read -r name cmd; do
            local type=$(classify_command "$cmd")
            classified_commands=$(echo "$classified_commands" | jq --arg cat "$type" --arg cmd "$cmd" --arg name "$name" \
                '.[$cat] = (.[$cat] // []) + [{"name": $name, "command": $cmd}]')
            discovered_commands+=("$cmd")
        done < <(python3 -c "
import json
with open('$project_root/package.json') as f:
    for k, v in json.load(f).get('scripts', {}).items():
        print(f'{k}\t{v}')
" 2>/dev/null)
    fi

    # Discover from Makefile
    if [ -f "$project_root/Makefile" ] || [ -f "$project_root/GNUmakefile" ]; then
        local mf="$([ -f "$project_root/Makefile" ] && echo "$project_root/Makefile" || echo "$project_root/GNUmakefile")"
        while IFS= read -r target; do
            [ -z "$target" ] && continue
            local cmd="make $target"
            local type=$(classify_command "$cmd")
            classified_commands=$(echo "$classified_commands" | jq --arg cat "$type" --arg cmd "$cmd" --arg target "$target" \
                '.[$cat] = (.[$cat] // []) + [{"name": $target, "command": $cmd}]')
            discovered_commands+=("$cmd")
        done < <(grep -oP '^[a-zA-Z0-9_-]+(?=\s*:)' "$mf" 2>/dev/null | grep -v '^\.' || true)
    fi

    # Store in state
    local cmds_json=$(printf '%s\n' "${discovered_commands[@]}" | jq -R . | jq -s .)
    echo "$cmds_json"
    echo "$classified_commands"
}

CI_COMMANDS=$(discover_and_store_ci_commands "$SPEC_PATH")
```

Then merge into state file:

```bash
jq --argjson ciCmds "$CI_COMMANDS" '.ciCommands = $ciCmds' \
   "$SPEC_PATH/.ralph-state.json" > "$SPEC_PATH/.ralph-state.json.tmp" && \
   mv "$SPEC_PATH/.ralph-state.json.tmp" "$SPEC_PATH/.ralph-state.json"
```

### 5.5 When CI Snapshot Is Captured

```
Timeline:
  T0: implement.md runs → discovers CI commands → stores in state
  T1: coordinator reads state → has ciCommands array
  T2: Before first task → capture pre-execution CI snapshot
      - For each "MUST PASS" category (test, build): run command, store result
      - For "SHOULD PASS" categories (lint, typecheck): run command, store result
  T3: Task 1 runs → verify passes → optional post-task CI (selective)
  T4: Spec completes → capture final CI snapshot → log to .progress.md
```

## 6. Edge Cases and Mitigations

### 6.1 No CI Commands Found

**Scenario**: Project has no `package.json`, `Makefile`, or CI config.

**Mitigation**:
- `ciCommands: []` in state — coordinator knows no auto-discovered CI exists
- Allow manual specification via state: `"manualCiCommands": ["echo check"]`
- Document this in the initial state output: "No CI commands discovered. If your project has CI, add it manually or run /ralph-specum:implement with --ci-commands"

### 6.2 CI Commands Require Setup

**Scenario**: `pytest` requires `source venv/bin/activate` first.

**Mitigation**:
- Store setup commands in a separate field: `"ciSetup": "source venv/bin/activate"`
- Run setup before each CI command: `eval "$setup_cmd && $ci_cmd"`
- Detect common setup patterns: `requirements.txt` → pip install, `package-lock.json` → npm ci

```bash
detect_setup_commands() {
    local root="$1"
    local setup=""

    if [ -f "$root/requirements.txt" ] && [ ! -d "$root/venv" ] && [ ! -d "$root/.venv" ]; then
        setup="${setup}pip install -r requirements.txt; "
    fi
    if [ -f "$root/package-lock.json" ]; then
        setup="${setup}npm ci; "
    fi
    if [ -f "$root/pnpm-lock.yaml" ]; then
        setup="${setup}pnpm install; "
    fi
    if [ -f "$root/Cargo.lock" ]; then
        setup="${setup}cargo fetch; "
    fi

    echo "$setup"
}
```

### 6.3 CI Commands Fail Before Any Task Runs

**Scenario**: Pre-execution CI snapshot shows `test: fail`.

**Mitigation**:
- Log to `.progress.md`: "CI baseline: test command failed before any tasks executed. This may indicate pre-existing issues."
- Continue execution — pre-existing CI failures are NOT a reason to stop
- Mark in snapshot: `"ciBaselineFailures": true` — coordinator uses this to distinguish "newly broken" from "already broken"

### 6.4 Metrics File Corruption

**Scenario**: Crash during write leaves incomplete JSON line.

**Mitigation**:
- On read, use `while IFS= read -r line; do echo "$line" | jq . >/dev/null 2>&1 || continue; done` to skip corrupt lines
- Or: `tail -n +$(grep -c '^{$' "$metrics_file")` to find last valid JSON object start
- The append-only pattern makes this rare; a single corrupt line is the worst case

### 6.5 Very Large Metrics Files

**Scenario**: Long-running specs accumulate thousands of entries.

**Mitigation**:
- No hard limit needed: JSONL is designed for this. Aggregation queries are O(n) but that's acceptable for post-hoc analysis.
- Optionally: rotate or compress after spec completion (out of scope for v1)
- Per-task metrics are lightweight (~500 bytes per entry); 1000 entries = ~500KB

### 6.6 Concurrent Writes (Parallel Execution)

**Scenario**: Multiple tasks writing metrics simultaneously.

**Mitigation**:
- flock already handles this (same pattern as chat.md in existing codebase)
- Each write is a single `cat >>` call; flock ensures mutual exclusion
- If flock fails (rare), the entry is lost but no corruption occurs

### 7.4 Key Implementation Notes

1. **Metrics file is per-spec, not global**. Each spec directory gets its own `.metrics.jsonl`. No cross-spec aggregation at this level.

2. **CI discovery runs once at state init**, not on every loop iteration. This avoids repeated command execution during the loop.

3. **Pre-execution CI snapshot is the baseline**. Everything compares against this. If no commands are discovered, snapshot is empty.

4. **The stop-watcher only reads metrics** — it checks for failure patterns in recent entries but never writes. This respects the stop-watcher's role as a read-only controller.

5. **Task-level verify commands remain independent of CI commands**. Task verify (per-task, scoped) and CI (project-wide, global) are separate tracking surfaces. Both must pass for completion.

6. **Metrics writing uses flock** — same concurrency model as the existing chat.md lock mechanism in spec-executor.md.

7. **Schema versioning**: The metrics JSON schema should include a `schemaVersion` field if the schema is expected to evolve. Start with `"schemaVersion": 1`.
