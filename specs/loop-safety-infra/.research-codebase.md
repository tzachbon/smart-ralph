## 1. stop-watcher.sh — Full Flow Analysis

**File**: `plugins/ralph-specum/hooks/scripts/stop-watcher.sh` (765 lines)

### Current Flow

**Step 1: Bootstrap (lines 1-38)**
- Reads hook input JSON from stdin for `cwd` and `transcript_path`
- Sources `path-resolver.sh` for multi-directory support
- Checks settings file `.claude/ralph-specum.local.md` for `enabled: false` (exit 0 if disabled)
- Resolves spec path via `ralph_resolve_current()`, extracts `SPEC_NAME` from path
- Opens state file at `$CWD/$SPEC_PATH/.ralph-state.json`

**Step 2: Race Condition Safeguard (lines 48-63)**
- Uses `stat` to check if state file was modified within last 2 seconds
- On Linux: `stat -c %Y`, on macOS: `stat -f %m`
- Sleeps 1 second if recently modified

**Step 3: ALL_TASKS_COMPLETE Detection (lines 65-259)**
- Parses transcript path from hook input
- Checks last 500 lines for `ALL_TASKS_COMPLETE` pattern (grep with word boundary)
- Fallback: checks last 20 lines for very recent signals
- On completion: updates epic state if `epicName` exists, runs `update-spec-index.sh --quiet`

**Step 4: Phase 4 Regression Sweep (lines 261-398)**
- Only triggers after ALL_TASKS_COMPLETE and only once per completion
- Parses `Dependency map` from `requirements.md` using awk
- Blocks loop, prompting coordinator to verify dependent specs' Observable signals and Hard invariants
- Outputs `REGRESSION_SWEEP_COMPLETE` after all sweeps

**Step 5: Phase 3 Repair Loop (lines 400-526)**
- Detects `VERIFICATION_FAIL`, `VERIFICATION_PASS`, `VERIFICATION_DEGRADED` in last 500 transcript lines
- **DEGRADED path**: Escalates to human (MCP Playwright not installed). Double-escalation guard checks for existing ESCALATE signal.
- **FAIL path**: Reads `repairIteration` from state (max 2 repairs). Classifies failure into 5 categories (impl_bug, env_issue, spec_ambiguity, flaky, test_quality). Generates REPAIR_REASON prompt. On VERIFICATION_PASS: resets repair state.
- On VERIFICATION_FAIL again: escalates on next iteration.

**Step 6: State Validation (lines 528-550)**
- Validates state file is readable JSON via `jq empty`
- Outputs BLOCK if corrupt

**Step 7: State Read and Global Iteration Check (lines 552-562)**
- Reads: `phase`, `taskIndex`, `totalTasks`, `taskIteration`, `quickMode`, `nativeSyncEnabled`
- Reads: `globalIteration`, `maxGlobalIterations` (default: 100)
- Blocks if `globalIteration >= maxGlobalIterations` (circuit breaker at global level)

**Step 8: Quick Mode Guard (lines 564-588)**
- If `quickMode: true` and phase != "execution": blocks stop, prevents user questions
- If `stop_hook_active: true`: allows stop to prevent loop

**Step 9: Role Boundaries / Field Validation (lines 590-693)**
- Reads baseline from `$SPEC_PATH/references/.ralph-field-baseline.json`
- Uses flock for file locking (`BASELINE_FILE.lock`)
- Iterates baseline fields, checks if agent-owned fields were modified
- In Phase 1: reports agent identity as "unknown"
- Type mismatch handling: skips fields where baseline type differs from state type

**Step 10: Execution Completion Cross-Check (lines 695-728)**
- When `taskIndex >= totalTasks`: verifies all tasks marked `[x]` in `tasks.md`
- If unchecked items remain: outputs BLOCK with detail

**Step 11: Loop Continuation (lines 730-817)**
- When `phase == execution` and `taskIndex < totalTasks`:
- Checks `awaitingApproval` — if true, allows stop for user gate
- Checks `stop_hook_active` — if true, skips to prevent re-invocation loop
- Extracts current task block from `tasks.md` using awk (handles [P] parallel groups)
- Builds REASON with task header, inline task block, and resume instructions
- Outputs BLOCK with decision/reason/systemMessage

## 2. path-resolver.sh — How It Works

**File**: `plugins/ralph-specum/hooks/scripts/path-resolver.sh` (253 lines)

### Core Functions

**`ralph_get_specs_dirs()`** (lines 44-111)
- Reads `specs_dirs` from YAML frontmatter of `.claude/ralph-specum.local.md`
- Format: `specs_dirs: ["./specs", "./packages/api/specs"]`
- Validates each path exists; warns and skips invalid paths
- Falls back to `./specs` if no valid paths

**`ralph_resolve_current()`** (lines 123-157)
- Reads `$RALPH_CWD/.claude/default-specs-dir/.current-spec` (default: `./specs/.current-spec`)
- Bare name (e.g., "loop-safety-infra") -> `$default_dir/$name`
- Full path (starts with `./` or `/`) -> used as-is
- Returns relative path string, not absolute

**`ralph_find_spec(name)`** (lines 162-217)
- Searches all configured specs_dirs for matching spec directory
- Exit 1 if not found, exit 2 if ambiguous (multiple matches)
- Returns full relative path on success

**`ralph_list_specs()`** (lines 221-252)
- Outputs `name|path` pairs across all configured directories
- Uses `find -maxdepth 1 -mindepth 1 -type d` to enumerate

### Key Integration Points
- Sets `RALPH_CWD` from hook input `cwd` (line 12: `RALPH_CWD="${RALPH_CWD:-$(pwd)}"`)
- All scripts source this file and set `RALPH_CWD` before calling functions
- Internal helpers: `_ralph_warn()`, `_ralph_validate_cwd()`, `_ralph_normalize_path()`

## 3. load-spec-context.sh — Spec Context Loading

**File**: `plugins/ralph-specum/hooks/scripts/load-spec-context.sh` (127 lines)

### Flow
1. Reads hook input for `cwd`
2. Sources `path-resolver.sh`
3. Checks settings file for `enabled: false` (exit if disabled)
4. Calls `ralph_resolve_current()` to get spec path
5. Reads `.ralph-state.json`: extracts `phase`, `taskIndex`, `totalTasks`, `awaitingApproval`
6. Outputs status summary to stderr
7. On non-execution phases: outputs next-command hints (requirements, design, tasks, implement)
8. On execution phase with awaitingApproval=false: suggests running `/ralph-specum:implement`
9. Reads `.progress.md` frontmatter for "Original Goal" and outputs it

### Baseline Capture (lines 110-124)
- On first run, creates `references/.ralph-field-baseline.json` with field ownership
- Captures field ownership: `chat.executor.lastReadLine`, `chat.reviewer.lastReadLine`, `external_unmarks`, `awaitingApproval`
- This baseline is later used by stop-watcher.sh for role boundary enforcement

## 4. implement.md — Execution Command

### ralph-specum version: `plugins/ralph-specum/commands/implement.md` (314 lines)

**Step 1: Resolve and Validate**
- Uses `ralph_resolve_current()` or `ralph_find_spec()` based on arguments
- Checks `tasks.md` exists
- Sets `$SPEC_PATH`

**Step 2: Parse Arguments**
- `--max-task-iterations` (default: 5)
- `--max-global-iterations` (default: 100)
- `--recovery-mode` (default: false)

**Step 3: Initialize State (lines 49-131)**
- Counts tasks: `grep -c -e '- \[.\]'` for total, `grep -c -e '- \[x\]'` for completed
- **CRITICAL: Merge into existing state** — reads existing state, merges with jq, writes temp file
- Preserves: `source`, `name`, `basePath`, `commitSpec`, `relatedSpecs`
- Sets 20+ execution fields including `nativeTaskMap`, `modificationMap`, `fixTaskMap`
- Backwards compatibility handling for missing fields

**Step 4: State Integrity Check (lines 136-156)**
- Drift detection: compares `taskIndex` against actual `[x]` count in `tasks.md`
- If `CURRENT_INDEX < COMPLETED`: corrects state (drift)
- If `CURRENT_INDEX > COMPLETED`: warns but doesn't correct

**Step 5: Parallel Reviewer Onboarding (lines 159-212)**
- Optional: creates `task_review.md` from template for external parallel reviewer
- Supports quality principles: SOLID, DRY, FAIL_FAST, TDD
- Writes FLOC-based coordination rules to `chat.md`

**Step 6: Coordinator Prompt (lines 218-275)**
- References: `coordinator-pattern.md`, `failure-recovery.md`, `verification-layers.md`, `phase-rules.md`, `commit-discipline.md`
- Key behaviors:
  - **CI snapshot separation**: Task verify and global CI reported separately
  - **Mechanical HOLD check**: grep `chat.md` for `[HOLD]`, `[PENDING]`, `[URGENT]`
  - **Read task_review.md**: Before every delegation
  - **Verification fabrications**: Never trust executor's claimed verification results
  - **VE task Skills validation**: Must have `e2e` or `playwright-env` in Skills field

**Step 7: Completion (lines 284-298)**
- Verifies all tasks `[x]` in `tasks.md`
- Deletes `.ralph-state.json`
- Cleans orphaned temp progress files
- Runs `update-spec-index.sh --quiet`
- Final git commit of spec files

### speckit version: `plugins/ralph-speckit/commands/implement.md` (1122 lines)

Same pattern but simpler: no multi-directory support, uses `.specify/specs/$feature/.speckit-state.json`, shorter coordinator prompt.

## 6. Existing Safety Patterns

### 6.1 Circuit Breaker Patterns

**Existing (partial):**
- **Global iteration limit** in stop-watcher.sh lines 563-566: if `globalIteration >= maxGlobalIterations`, blocks execution
- **Max task iterations**: per-task retry limit (checked by coordinator)
- **Repair iteration limit**: max 2 repair attempts per story (stop-watcher.sh lines 501-502, 532-533)
- **Max fix task depth/count**: tracked in fixTaskMap with depth checking

**Missing (per gap analysis):**
- No per-consecutive-failure circuit breaker (only global iteration count)
- No time-based circuit breaker (48h rule from plan)
- No metrics-driven circuit breaker

### 6.2 Git Operations

**Patterns found:**

| Location | Pattern | Purpose |
|----------|---------|---------|
| implement.md:294-295 | `git add tasks.md + git commit` | Final progress update |

## 9. Key Design Patterns for Implementation

### Pattern 1: Graceful Degradation
All state reads use fallback defaults: `jq -r '.field // default' file || echo "default"`. This handles:
- Missing state files
- Missing fields (backwards compatibility)
- Corrupt JSON (validated before use)

### Pattern 2: JSON Output Blocks
All hooks output JSON via `jq -n --arg reason X --arg msg Y '{decision, reason, systemMessage}'`. This is the standard interface for Claude Code hooks.

### Pattern 3: Race Condition Mitigation
- File age check before reading (stat-based, 2-second threshold)
- Retry loop for jq parsing (3 attempts, 1-second delay)
- flock for concurrent baseline writes
- Temp file + mv for atomic state writes

### Pattern 4: Transcript-Based Signal Detection
The stop-watcher uses transcript grep (not state parsing) to detect:
- ALL_TASKS_COMPLETE (completion signal)
- VERIFICATION_FAIL/PASS/DEGRADED (verification signals)
- ESCALATE (human escalation)

### Pattern 5: Multi-Plugin Architecture
Three stop-watcher implementations:
1. `plugins/ralph-specum/hooks/scripts/stop-watcher.sh` — Full featured (765 lines)
2. `plugins/ralph-speckit/hooks/scripts/stop-watcher.sh` — Simplified (193 lines)
3. `plugins/ralph-specum-codex/hooks/stop-watcher.sh` — Minimal (62 lines)

### Pattern 6: Safety Infrastructure Integration Points

For the loop-safety-infra spec, the recommended integration points are:

| Safety Feature | Primary Integration Point | Secondary |
|---------------|--------------------------|-----------|
| Pre-loop git checkpoint | `commands/implement.md` Step 3 | `stop-watcher.sh` pre-loop check |
| Circuit breaker | `stop-watcher.sh` (new section) | `commands/implement.md` (state init) |
| Per-task metrics | `commands/implement.md` (coordinator prompt) | `stop-watcher.sh` (append on completion) |
| Read-only detection | `commands/implement.md` Step 1 | `stop-watcher.sh` (heartbeat check) |
| CI snapshot tracking | `commands/implement.md` (coordinator) | `schemas/spec.schema.json` (add ciCommands) |

### Pattern 7: Adding New State Fields
When adding new safety fields:
1. Add to `schemas/spec.schema.json` in the `state.properties` section
2. Initialize in `commands/implement.md` Step 3 (merge section)
3. Document in `role-contracts.md` State Field Ownership table
4. Update baseline in `references/.ralph-field-baseline.json`
