# Progress: codex-plugin-sync

## Goal
Make sure that the existing codex platform files (platforms/codex/) aren't out of date with the claude ralph-specum plugin, and introduce a real codex plugin based on the OpenAI Codex plugin spec (https://developers.openai.com/codex/plugins).

## Learnings
- Codex plugin entry point is `.codex-plugin/plugin.json` (not `.claude-plugin/`). Only this file goes in `.codex-plugin/`; everything else (skills/, .app.json, .mcp.json) lives at plugin root.
- Skills use the open Agent Skills standard (agentskills.io) — SKILL.md format is cross-platform (Codex, Claude Code, Gemini CLI, Cursor). The 15 existing skills in `platforms/codex/skills/` require no format changes.
- Codex has no slash commands. Claude Code's `/ralph-specum:start` becomes `$ralph-specum-start` (dollar-sign skill invocation).
- The Stop hook (`decision: "block"` + `reason`) is the Codex equivalent of Ralph's stop-watcher. CRITICAL: it is experimental, behind `[features] codex_hooks = true`, and disabled on Windows. High-risk dependency for the execution loop.
- `platforms/codex/` is v4.8.4; Claude plugin is v4.9.1. Need to audit delta before syncing.
- Codex has no agents/ markdown definitions. Agent behavior must be encoded in skill bodies or spawned via `spawn_agent` tool (max nesting depth 1, max 6 concurrent threads).
- Plugin marketplace entry goes in `$REPO_ROOT/.agents/plugins/marketplace.json`. Install policy options: `INSTALLED_BY_DEFAULT`, `AVAILABLE`, `NOT_AVAILABLE`.
- No package.json / build step in this repo. No automated E2E tooling. Verification is manual: install plugin, invoke skills.
- `$plugin-creator` built-in skill can scaffold the new plugin structure automatically inside Codex.
- Progressive disclosure in skills: name+description pre-loaded at startup (level 1), full SKILL.md loaded on demand (level 2), referenced files loaded on demand (level 3). Keep SKILL.md under 2000 words.

- Chosen approach: Codex-native plugin with skills as primary interface, agent configs as bootstrap templates, Stop hook for execution loop with fallback docs.

## Learnings (Design Phase)
- All 3 "net-new" agents (spec-reviewer, qa-engineer, refactor-specialist) actually have markdown sources in `plugins/ralph-specum/agents/`. No authoring from scratch needed; adapt from existing Claude agents.
- Agent bootstrap delivery must be `.toml.template` files (not live `.toml`), because Codex agents are `config.toml` keys -- plugins cannot inject into user config automatically.
- Stop hook output for Codex is `{"decision": "block", "reason": "..."}` -- different from Claude's continuation prompt pattern. Stop-watcher rewrite is non-trivial but the logic is the same.
- Missing content (verification-layers, failure-recovery, intent-classification references) should be merged into `workflow.md` rather than adding 11 new reference files -- keeps the reference count manageable.
- `tasks.md` and `settings-template.md` need full replacement (not merge) since the Claude versions are 3-4x longer with structural additions.
- Version sync CI can be a single shell script (`tests/helpers/version-sync.sh`) comparing 3 JSON fields -- no new CI workflow file needed if added to existing `codex-version-check.yml`.
