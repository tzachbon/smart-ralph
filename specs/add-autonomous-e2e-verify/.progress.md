# Progress: add-autonomous-e2e-verify

## Original Goal

Add autonomous end-to-end verification to the task planning phase. When tasks are generated, the system should:

1. Ask the user (or auto-decide in quick mode) whether to add autonomous verification tasks at the end
2. Research existing tooling for automated checking (browsers, simulators, etc.)
3. Add verification tasks that spin up whatever infrastructure is needed to test the actual output end-to-end
4. If verification fails, add more tasks to fix issues and re-verify in a loop
5. In quick mode: autonomous verification defaults to YES
6. In normal mode: ask the user (default YES), suggest verification approach, let user customize

Examples: spin up browser for websites, mobile simulator for apps, check mailbox for sent emails, etc.

## Learnings
- VE tasks should NOT create a new phase -- they extend the existing final verification sequence (V4/V5/V6 -> VE1/VE2/VE3) to avoid restructuring all phase numbering
- Existing recovery mode (fixTaskMap, fix task generation, retry) is fully reusable for VE verify-fix-reverify -- no new loop mechanism needed
- qa-engineer already handles [VERIFY] tasks; VE tasks are standard [VERIFY] tasks, no new agent required
- Research phase already has flexible topic structure; adding "Verification Tooling" as a research topic is straightforward
- Key risk: VE-cleanup must be bulletproof to avoid orphaned processes. Cleanup should use `kill` by PID + `lsof -ti :PORT | xargs kill` as fallback
- Task planner already has "Fully Autonomous = End-to-End Validation" section but it's aspirational guidance, not enforced via dedicated tasks. VE tasks make it concrete
- The tasks command (tasks.md) already has interview step 2 and approach proposals -- VE prompt fits naturally into the interview flow
- VE tasks must always be sequential (infrastructure state dependency) -- [P] tag never applies
- Coordinator needs ONE addition: VE-cleanup guarantee (skip-to-cleanup on max retries instead of hard stop) -- all other VE handling uses existing [VERIFY] + recovery mode paths
- PID storage uses /tmp/ve-pids.txt (not state file) -- simple, cleanup removes it, no state pollution
- Research-analyst verification tooling discovery mirrors existing Quality Commands pattern (table format, detection via package.json/configs)
