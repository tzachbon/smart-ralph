# Progress: bmad-bridge-plugin

## Phase: research (complete)

## Learnings
- BMAD is NOT a Claude Code plugin — it is a standalone npm package (bmalph v2.11.0) with its own plugin system at `~/.bmad/cache/external-modules/`
- BMAD output artifacts are markdown files in configurable `_bmad-output/` subdirectories with YAML frontmatter
- BMAD PRD (`prd.md`) has NO fixed template beyond frontmatter — content is built interactively through 12 collaborative steps; functional requirements use `- FR#: [Actor] can [capability]` format
- BMAD epics (`epics.md`) is a monolithic document containing overview + requirements inventory + epic list + ALL stories in one file
- BMAD architecture (`architecture.md`) has NO fixed structure — sections appended collaboratively through 8 steps
- BMAD stories use Given/When/Then BDD acceptance criteria format
- BMAD sprint status is a YAML file with `development_status` mapping (epic-1: backlog, 1-1-story-key: ready-for-dev, etc.)
- The existing `ralph_import.sh` in bmalph is DEPRECATED — it uses Claude Code LLM to convert PRDs, not a structural mapper
- Legacy Ralph used `.ralph/` directory (PROMPT.md, @fix_plan.md) — completely different from smart-ralph's `specs/` directory
- NO existing BMAD→smart-ralph structural mapper exists in the repo
- Smart-ralph spec templates use YAML frontmatter with `spec`, `phase`, `created` keys and have very specific section structures
- BMAD FR format `- FR#: [Actor] can [capability]` maps cleanly to user stories: Actor→As a, capability→I want, benefit derived from context
- BMAD epics.md has a dual epic list structure (Requirements Inventory FR Coverage Map + detailed Epic N sections with stories) — both must be parsed
- BMAD architecture.md ADRs are prose-based, not structured — must pattern-match headings like "decisions", "architecture", "technology" rather than relying on ADR convention
- Task-planner tasks.md template uses POC-first workflow for greenfield specs, with Quality Checkpoints every 2-3 tasks and VE tasks for e2e verification
- Requirements.md template includes Risks table and Verification Contract section not present in design.md or tasks.md templates
- Architecture choice: single monolithic import.sh with inline bash functions (not separate module files) — keeps <500 line constraint and avoids module overhead
- PRD parsing strategy: awk state-machine for robust section boundary detection (single pass, no multiple greps)
- Epics parsing: state-machine tracking current epic/story context when encountering `### Story N.M:` headings
- Architecture parsing: heading pattern-matching (not ADR convention) since BMAD architecture.md has no fixed template
- Error strategy: graceful degradation — missing PRD sections warn but don't fail; missing artifacts generate minimal valid output
- Hard invariant enforced: target spec directory must NOT already exist — fail with error on overwrite attempt
- Testing approach: inline bash test harness (no npm test runner in repo; fixtures as inline heredocs)
- Generated tasks.md will include full Phase 1-5 template structure with Phase 1 populated from stories, other phases as empty template sections
- Task-planner generated 50 tasks for fine-grained POC-first workflow (target 40-60+)
- POC phase (1.1-1.23): 23 tasks — all import.sh functions built incrementally, CLI wrapper, then 3 POC checkpoints
- Refactoring phase (2.1-2.10): 10 tasks — extract helpers, standardize error handling, input sanitization, path traversal protection
- Testing phase (3.1-3.9): 9 tasks — test harness infrastructure, unit tests for each parser function, integration tests for full flow and error paths
- Quality Gates phase (4.1-4.2): 2 tasks — syntax/lint/line-count checks, PR creation
- PR Lifecycle phase (5.1-5.3): 3 tasks — CI monitoring, review resolution, final validation
- VE tasks: 3 tasks — E2E import against mini BMAD fixture, FR mapping accuracy check, cleanup
- Each VE task includes `Skills: e2e` metadata for executor skill loading
- No Playwright/UI VE tasks needed (CLI project type — no UI present)
- Test doubles: all testing uses fixture files (inline heredocs), no mocks or fakes needed per design.md test strategy
